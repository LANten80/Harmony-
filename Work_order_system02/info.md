# **JS鸿蒙应用开发实战项目详细计划书**

## **一、项目概述**

### 1.1 项目名称
**工单管理应用（HarmonyOS版）**

### 1.2 项目背景与意义
随着企业数字化转型进程加速，工单管理系统成为提升运营效率、优化服务流程的核心工具。传统工单管理系统多基于Web或原生移动应用，存在跨平台兼容性差、开发维护成本高等问题。本项目基于HarmonyOS分布式操作系统，采用ArkTS声明式开发范式，旨在打造一款高效、直观、跨设备协同的现代化工单管理应用。

项目具有以下重要意义：
1. **技术探索价值**：深入实践HarmonyOS应用开发全流程，掌握新一代操作系统开发理念
2. **教学示范价值**：作为软件工程专业实践教学案例，展示完整项目开发方法论
3. **商业应用价值**：为中小企业提供轻量级、可定制的工单管理解决方案
4. **生态建设价值**：丰富HarmonyOS应用生态，探索企业级应用开发模式

### 1.3 项目核心目标
- **功能完整性**：实现工单全生命周期管理（创建、查看、编辑、追踪、统计）
- **技术先进性**：全面应用ArkTS声明式UI和状态管理机制
- **架构规范性**：采用分层架构设计，确保代码可维护性和可扩展性
- **体验优秀性**：提供流畅自然的用户交互，适配不同设备形态
- **数据一致性**：确保多组件间数据同步和状态一致性

### 1.4 项目特色与创新点
1. **声明式开发范式**：完全采用ArkTS声明式语法，提升开发效率和代码可读性
2. **状态驱动UI**：基于状态变化自动更新UI，简化数据绑定逻辑
3. **组件化架构**：高度模块化的组件设计，支持功能扩展和复用
4. **跨设备协同**：利用HarmonyOS分布式能力，支持多设备间工单状态同步
5. **渐进式交互**：流畅的动画过渡和渐进式交互反馈，提升用户体验

### 1.5 技术架构选型
| 技术层级 | 技术选择 | 版本/标准 | 选型理由 |
|---------|---------|----------|---------|
| **操作系统** | HarmonyOS | 4.0+ | 分布式架构、跨设备协同、国产化支持 |
| **开发语言** | ArkTS | TypeScript超集 | 类型安全、生态成熟、声明式语法 |

1. **分层解耦原则**：清晰划分表示层、业务逻辑层、数据层
2. **组件化原则**：功能模块高度内聚，接口清晰定义
3. **状态驱动原则**：UI随状态变化自动更新，减少手动操作
4. **可扩展性原则**：支持功能模块的灵活添加和替换

### 3.2 状态管理架构

#### 3.2.1 状态管理策略
本项目采用ArkTS内置的状态管理装饰器实现多层次的状态管理：

1. **页面级状态 (@State)**
   - 管理页面内部的状态变化
   - 如：弹窗显示状态、列表选中状态

2. **组件级状态 (@State/@Prop)**
   - 组件内部状态使用@State
   - 父向子传递状态使用@Prop

3. **父子组件双向同步 (@Link)**
   - 父组件和子组件共享状态
   - 状态变更双向同步

4. **跨组件状态共享 (@Provide/@Consume)**
   - 跨层级组件状态共享
   - 避免属性逐层传递

5. **状态变化监听 (@Watch)**
   - 监听特定状态变化
   - 执行相应业务逻辑

#### 3.2.2 数据流向设计
```
用户操作
    ↓
触发UI事件
    ↓
修改@State状态
    ↓
@Watch监听状态变化
    ↓
调用DataModel操作方法
    ↓
更新数据存储
    ↓
触发@Provide状态变更
    ↓
@Consume组件接收更新
    ↓
自动更新UI渲染
```

### 3.3 组件设计规范

#### 3.3.1 组件分类
1. **容器组件**
   - 负责布局和容器管理
   - 如：MainPage、TargetList

2. **展示组件**
   - 负责数据展示和呈现
   - 如：TargetInformation、TargetListItem

3. **交互组件**
   - 负责用户交互和反馈
   - 如：ProgressEditPanel、AddTargetDialog

4. **功能组件**
   - 提供特定功能服务
   - 如：Loading、Toast、Dialog

#### 3.3.2 组件通信机制
1. **属性传递 (@Prop)**
   - 父组件向子组件传递数据
   - 单向数据流，子组件不能直接修改

2. **事件回调**
   - 子组件向父组件传递事件
   - 通过回调函数实现

3. **共享状态 (@Provide/@Consume)**
   - 跨层级组件共享状态
   - 状态变更自动通知所有消费者

4. **全局状态管理**
   - 通过单例模式管理全局状态
   - 如：用户信息、主题设置

### 5.2 关键状态管理场景

#### 5.2.1 列表项互斥展开场景
**问题描述**：需要实现点击一个工单项时，其他已展开的工单项自动折叠

**解决方案**：
1. 在TargetList中使用@State管理当前点击的索引clickIndex
2. 通过@Link将clickIndex传递给每个TargetListItem
3. 在TargetListItem中使用@Watch监听clickIndex变化
4. 当clickIndex不等于当前项索引时，自动折叠当前项

**状态流转**：
```
用户点击ListItem A
    ↓
设置clickIndex = A的索引
    ↓
@Link同步到所有ListItem
    ↓
ListItem B的@Watch触发
    ↓
判断clickIndex ≠ B的索引
    ↓
自动折叠ListItem B
    ↓
ListItem A保持展开状态
```

#### 5.2.2 整体进度联动更新场景
**问题描述**：当某个工单项进度更新为100%时，需要实时更新整体进度统计

**解决方案**：
1. 在MainPage中使用@Provide提供全局刷新状态overAllProgressChanged
2. 在TargetListItem中使用@Consume消费该状态
3. 工单项进度更新后，修改overAllProgressChanged触发重新计算
4. MainPage中的@Watch监听变化，重新计算统计数据

**状态流转**：
```
用户点击"完成"按钮
    ↓
设置进度为100%
    ↓
调用DataModel.updateProgress()
    ↓
修改overAllProgressChanged状态
    ↓
@Provide通知所有@Consume组件
    ↓
MainPage的@Watch触发
    ↓
重新计算总进度和完成数
    ↓
更新TargetInformation显示
```

#### 5.2.3 弹窗数据回调场景
**问题描述**：弹窗中输入的数据需要回传到主页面进行处理

**解决方案**：
1. 在AddTargetDialog中定义回调函数onClickOk
2. 在MainPage中创建弹窗时传入saveTask方法作为回调
3. 弹窗中点击确定时调用onClickOk回调
4. 回调函数中处理数据保存逻辑

**数据流转**：
```
MainPage打开AddTargetDialog
    ↓
传入saveTask作为onClickOk回调
    ↓
用户在弹窗中输入内容
    ↓
点击确定按钮
    ↓
调用this.onClickOk(inputValue)
    ↓
执行MainPage的saveTask方法
    ↓
验证并保存数据到DataModel
    ↓
更新页面状态
    ↓
关闭弹窗
```

## **六、数据模型设计**

### 6.1 实体模型设计

#### 6.1.1 工单实体模型 (TaskItemModel)
**核心属性定义**：
1. **基础信息**
   - id: 工单唯一标识（UUID生成）
   - taskName: 工单名称（必填，长度限制）
   - description: 工单描述（可选）

2. **进度信息**
   - progressValue: 进度值（0-100整数）
   - status: 状态枚举（未开始、进行中、已完成）
   - priority: 优先级（低、中、高）

3. **时间信息**
   - createDate: 创建时间（ISO格式）
   - updateDate: 更新时间（ISO格式）
   - dueDate: 截止时间（可选）

4. **关联信息**
   - assignee: 负责人（可选）
   - category: 分类标签（可选）
   - tags: 标签数组（可选）

**状态计算规则**：
- progressValue = 0 → 状态为"未开始"
- 0 < progressValue < 100 → 状态为"进行中"
- progressValue = 100 → 状态为"已完成"

#### 6.1.2 数据操作模型 (DataModel)
**核心功能设计**：
1. **数据存储管理**
   - 使用数组存储工单数据
   - 提供数据深拷贝接口
   - 支持数据持久化扩展

2. **CRUD操作接口**
   - 增：addData() 添加新工单
   - 查：getData() 获取所有工单
   - 改：updateProgress() 更新进度
   - 删：deleteTask() 删除工单

3. **数据验证逻辑**
   - 输入参数合法性验证
   - 数据完整性检查
   - 业务规则校验

4. **统计计算功能**
   - 计算总体统计数据
   - 按分类统计
   - 按状态统计
   - 按时间统计

5. **监听通知机制**
   - 数据变更监听器注册
   - 变更事件通知
   - 错误处理回调
